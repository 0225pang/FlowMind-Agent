一、项目基本信息

项目名称：
FlowMind Agent：基于飞书生态的 AI 内容运营与知识管理智能体平台

项目定位：
本系统是一个面向内容运营、教育服务、个人 IP 运营和小型团队协作场景的 AI 智能体平台。系统以网页端 AI 对话工作台作为统一入口，结合飞书生态中的文档、多维表格、任务协作、群机器人等能力，并结合本地业务数据库、向量数据库和大模型 API，实现内容选题生成、资料库整理、知识问答、学员信息管理、院校项目情报管理、数据可视化和任务协同。

系统核心思想：
把原本依靠人工经验完成的“选题生成、文案撰写、资料整理、知识沉淀、学员分析、院校信息跟踪、飞书协作”流程，沉淀为可复用的智能体工作流和业务数据资产。

系统产品形态：
后台管理系统 + AI 工作台结合的网页端平台。用户不需要手动选择多个分散工具，而是通过一个统一 AI 工作台输入自然语言任务，由系统自动判断应该调用内容智能体、知识库智能体、学员分析智能体、院校情报智能体或飞书同步智能体。

三、用户需求与问题分析

请详细描述系统服务的用户类型、用户需求和痛点。

用户类型至少包括：

1. 内容运营人员
   需求：
   - 快速生成保研相关小红书选题。
   - 生成朋友圈专业人设文案。
   - 管理历史文案、主题库和内容日历。
   - 查看某个主题下生成过哪些文案。
   - 知道某条文案是否已使用、在哪一天使用。
   - 对文案和主题进行评分，判断重要性或受欢迎程度。
   - 对生成内容进行编辑和人工复核。
   痛点：
   - 人工写作耗时。
   - 爆款结构依赖经验。
   - 历史内容散落在聊天记录、飞书文档和本地文件中。
   - 内容发布计划缺乏统一视图。

2. 教育服务老师 / 保研咨询老师
   需求：
   - 管理学员 GPA、排名、英语成绩、目标院校、申请阶段。
   - 快速分析学员风险等级。
   - 根据学生画像推荐院校项目。
   - 查询保研知识库、院校通知、项目要求。
   痛点：
   - 学员信息分散。
   - 院校项目截止时间多、材料复杂。
   - 依靠人工经验容易遗漏风险点。

3. 个人 IP 运营者 / 教育博主
   需求：
   - 基于专业人设生成朋友圈文案。
   - 基于知识库沉淀爆款标题模板、话术库、内容 SOP。
   - 让内容表达既专业可信，又不显得过度营销。
   痛点：
   - 内容表达需要兼顾专业性和转化效果。
   - 缺乏稳定的内容生产系统。

4. 小型团队管理员
   需求：
   - 管理团队成员权限。
   - 配置模型、Prompt、飞书应用能力。
   - 维护系统日志和同步状态。
   - 让组员扩展新能力时尽量不影响主流程。
   痛点：
   - 多人协作容易改乱核心代码或业务流程。
   - 缺少统一的扩展规范。

5. 答辩/评审用户
   需求：
   - 看到完整系统边界。
   - 看到 AI 工作台不仅是聊天框，而是业务系统入口。
   - 看到明确的领域模型、用例图、包图、类图和动态流程。
   痛点：
   - 普通 Demo 容易显得空壳。
   - 需要证明系统具有可扩展性和业务完整性。

四、竞品分析

请写竞品分析，至少比较以下类型：

1. 通用 AI 助手，例如豆包、ChatGPT、通义千问
   优势：
   - 生成能力强。
   - 对话体验成熟。
   - 可泛化处理多类任务。
   不足：
   - 不直接绑定本地业务数据。
   - 不天然管理主题库、文案库、内容日历。
   - 不自动沉淀到飞书文档和多维表格。
   - 缺少针对保研教育服务场景的业务结构。

2. 飞书文档 / 飞书多维表格
   优势：
   - 协作能力强。
   - 文档和数据管理成熟。
   - 适合团队沉淀资料。
   不足：
   - 内容生成、知识检索和业务 SOP 需要人工组织。
   - AI 能力不是以业务智能体工作流为核心。
   - 多个表格和文档之间仍需要人工维护。

3. Notion AI / 知识管理工具
   优势：
   - 知识库体验好。
   - 文档组织灵活。
   不足：
   - 对中文教育服务和保研场景不够贴合。
   - 与飞书生态和国内团队协作习惯结合较弱。
   - 内容运营数据化管理不足。

4. 传统后台管理系统
   优势：
   - 数据管理清晰。
   - 权限和表单流程规范。
   不足：
   - 缺少自然语言交互。
   - 缺少智能体自动路由。
   - 缺少内容生成和知识检索能力。

请总结 FlowMind Agent 的优势：
- 统一 AI 工作台作为入口。
- 结合内容运营、知识管理、飞书协作和学员/院校业务。
- 支持内容资产沉淀，不只是一次性生成。
- 支持工具调用过程和模型思考过程记录，提升可追溯性。
- 采用模块化服务边界，便于后续扩展 MCP、Skill、飞书 API、向量数据库和不同大模型。

请也写劣势：
- Demo 阶段部分飞书能力仍为 Mock 或半真实调用。
- 生产级权限、审计、审批、部署能力仍需完善。
- 向量检索质量依赖知识库数据质量。
- AI 生成内容仍需要人工复核。

3. AI 工作台模块
   - 统一自然语言输入框
   - 默认总智能体，不要求用户手动选择智能体
   - 自动识别任务类型
   - 自动路由到以下智能体：
     a. 内容选题与文案智能体 ContentAgent
     b. 知识库整理与问答智能体 KnowledgeAgent
     c. 学员分析智能体 StudentAgent
     d. 院校情报智能体 SchoolAgent
     e. 飞书同步智能体 FeishuAgent
   - 支持流式回复

   - 支持模型 Thinking / Reasoning 展示
   - 支持工具调用过程 Trace 展示
   - 支持会话历史
   - 支持上下文面板
   - 支持快捷 Prompt
   - 支持“生成选题”“整理资料”“分析学员”“推荐院校”“飞书同步”等快捷动作
   - 工具调用只展示实际调用的工具，不展示未调用工具
   - 工具结果可展开查看

4. 内容运营模块
   注意：小红书 SOP 和朋友圈 SOP 的生成入口主要放在 AI 工作台中，内容运营模块主要用于管理生成后的内容资产。
   - 主题库
     - 展示所有主题
     - 3×2 卡片布局
     - 分页
     - 手动新增主题
     - 删除主题
     - 主题评分，使用 5 星评价
     - 标签、平台、状态、热度、计划日期
   - 文案库
     - 展示所有文案
     - 分页
     - 手动新增文案
     - 删除文案
     - 编辑文案
     - 文案评分，使用 5 星评价
     - 查看文案是否已使用
     - 查看文案在哪一天使用
     - 支持文案配图
     - 如果没有图，则显示配图建议
   - 内容日历
     - 保持日历表设计
     - 有内容发布的日期加标记
     - 点击有内容的日期展开对应内容列表
     - 展示发布内容、平台、状态
   - 历史文案
     - 点击某个主题卡片，查看该主题下生成过的历史文案
     - 可编辑历史文案
     - 可上传/关联图片
     - 可查看使用状态和使用日期

5. 内容智能体 SOP 需求
   小红书内容智能体 SOP：
   - 输入：主题、目标人群、风格
   - Step 1：爆款结构检索
   - Step 2：结构压缩成模板
   - Step 3：仿写但不抄袭
   - Step 4：生成 10 条标题
   - Step 5：输出 3 个版本：干货版、情绪增强版、转化引导版
   - Step 6：自动入库到内容库和飞书沉淀结构

   朋友圈内容智能体 SOP：
   - 输入：场景、身份标签、情绪方向
   - Step 1：场景识别
   - Step 2：人设映射
   - Step 3：朋友圈结构生成
   - Step 4：生成 3 种风格：专业理性版、学姐温和版、稍带传播版
   - Step 5：自动优化，删除营销感词汇，增强真实细节，控制长度 100-300 字
   - Step 6：入库飞书和本地文案库

   知识库/内容资产智能体 SOP：
   - 输入：已生成内容、小红书笔记、朋友圈内容、招生话术、SOP
   - 自动分类
   - 结构拆解
   - 生成可复用模板库
   - 自动打标签
   - 写入飞书多维表格或本地数据库
   - 建立向量索引，方便后续检索

十二、动态模型

请至少抽取 3 个系统用例，绘制活动图、顺序图或状态机图。

用例1：AI 工作台问答流程
步骤：
- 用户输入问题。
- 前端发送流式请求。
- 服务端保存用户消息。
- AgentRouter 判断任务类型。
- 系统优先检索向量知识库。
- 系统根据任务调用飞书、时间、联网、内容SOP等工具。
- 工具调用结果形成 Trace。
- LLM 生成模型 Thinking 和最终回答。
- 前端流式展示 Thinking、Trace 和回答。
- 服务端保存助手消息、工具Trace和Thinking。
- 用户切换会话后仍可查看历史记录。

请绘制顺序图。

用例2：小红书内容生成与入库流程
步骤：
- 用户输入主题。
- 系统识别为内容生成任务。
- 检索爆款结构或历史模板。
- 生成结构模板。
- 生成标题、正文和多个版本。
- 生成标签和配图建议。
- 保存到主题库和文案库。
- 可同步飞书文档或多维表格。
- 用户在内容运营页面查看、评分、编辑和安排日历。

请绘制活动图。

用例3：知识库同步与向量检索流程
步骤：
- 用户触发飞书知识库同步。
- 系统读取飞书文档元数据和正文。
- 保存到本地知识库。
- 生成摘要和标签。
- 文本切分为 Chunk。
- 调用 Embedding 服务生成向量。
- 写入 Weaviate。
- 用户提问时优先向量检索。
- 如果未命中，回退到 MySQL 搜索。
- AI 回答中说明资料来源或未命中情况。

请绘制活动图或顺序图。

可选用例4：学员画像分析与院校推荐
步骤：
- 用户选择学员。
- 系统读取学员 GPA、排名、英语成绩、目标院校、申请阶段。
- 系统判断风险等级。
- 查询院校项目要求。
- 计算匹配度。
- AI 生成申请建议。
- 前端展示分析结果。

请绘制顺序图或活动图。

十三、非功能性需求

请补充非功能性需求：

1. 可用性
   - 页面适配 16:9 展示。
   - AI 工作台响应式布局。
   - 工具调用详情展开后不影响整体滚动。
   - 表格、卡片、日历交互清晰。

2. 可维护性
   - 前后端分离。
   - 后端模块边界清晰。
   - 新能力通过 ToolService、Controller、Extension 或 MCP/Skill 扩展。
   - 避免所有逻辑堆在一个文件。

3. 可扩展性
   - LLM API 可替换。
   - 飞书 API 可从 Mock 切换到真实接口。
   - 向量数据库可替换或扩展。
   - 后续可拆分为真实微服务。

4. 安全性
   - API Key 不进入 Git 仓库。
   - 敏感配置放本地配置或环境变量。
   - 用户访问需要认证。
   - 后续引入 RBAC 权限模型。
   - 工具调用采用白名单，不允许 LLM 任意执行 Shell。

5. 数据一致性
   - 会话历史、工具 Trace、模型 Thinking 要持久化。
   - 内容主题、文案、日历之间要保持关联。
   - 知识库文档与向量索引要保留来源 token。

6. 可观测性
   - 系统日志。
   - 飞书同步日志。
   - 工具调用记录。
   - AI 会话记录。
   - 向量检索来源记录。
