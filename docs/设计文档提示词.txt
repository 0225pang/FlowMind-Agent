一、项目基本信息

项目名称：
FlowMind Agent：基于飞书生态的 AI 内容运营与知识管理智能体平台

项目定位：
FlowMind Agent 是一个面向内容运营、教育服务、个人 IP 运营和小型团队协作场景的 AI 智能体平台。系统以网页端 AI 对话工作台作为统一入口，结合飞书生态中的文档、多维表格、任务协作和群机器人能力，并结合本地业务数据库、向量数据库和大模型 API，实现内容选题生成、资料库整理、知识问答、学员信息管理、院校项目情报管理、数据可视化和任务协同。

系统产品形态：
前端是 Vue3 + TypeScript + Vite + Element Plus 的网页端应用，页面形态为“后台管理系统 + AI 工作台”。后端是 Spring Boot 3.x 多模块 Maven 工程。Demo 阶段采用 app-service 聚合启动，其余模块体现微服务边界；后续可以拆分成真实微服务。

系统核心能力：
1. AI 工作台统一入口
2. Agent Router 自动路由
3. ContentAgent、KnowledgeAgent、StudentAgent、SchoolAgent、FeishuAgent
4. 小红书内容 SOP、朋友圈内容 SOP、内容资产沉淀 SOP
5. 内容主题库、文案库、内容日历
6. 知识库文档同步与向量检索
7. 飞书文档、飞书共享文件夹、飞书同步 Mock 与真实接口预留
8. 学员管理、院校情报、数据分析
9. SSE 流式输出
10. 模型 Thinking / Reasoning 展示与落库
11. 工具调用 Trace 展示与落库
12. MySQL 本地业务数据库
13. Weaviate 向量数据库
14. Redis、MinIO 预留
15. Swagger/OpenAPI 文档

九、服务协作与事务设计

请详细说明不同服务之间如何协作。

场景1：AI 工作台问答
- 前端发送 SSE 请求。
- ai-agent-service 保存用户消息。
- AgentRouter 判断任务。
- AgentTraceService 收集工具上下文。
- AgentExtension 调用向量检索、飞书工具、时间工具、联网工具等。
- LLMClient 调用大模型。
- 后端流式返回 reasoning 和 answer。
- 保存 assistant 消息、Trace、Thinking。

请说明同步关系：
- 用户问答是同步请求 + 流式响应。
- 工具调用需要在模型生成前完成，作为上下文注入。
- LLM 生成过程通过 SSE 返回。

场景2：内容生成与入库
- AI 工作台触发内容 SOP。
- content-service 保存主题和文案。
- content_calendar 记录发布日期。
- 可选同步到飞书文档或多维表格。
- 生成流水记录保存输入输出摘要。

场景3：知识库同步与向量检索
- feishu-service 或 knowledge-service 读取飞书文档。
- knowledge-service 保存文档。
- EmbeddingService 生成向量。
- WeaviateClient 写入向量对象。
- 查询时先向量检索，失败时 MySQL fallback。

跨服务事务设计：
- Demo 阶段以单体聚合方式运行，可依赖本地事务。
- 未来真实微服务拆分后，跨服务不建议使用分布式强事务。
- 可采用最终一致性：
  - 本地事务先保存主业务数据。
  - 生成同步任务或事件。
  - 下游服务异步处理。
  - 同步记录表保存状态。
  - 失败可重试。
- 对外部系统调用，如飞书写入，不应和本地数据库强绑定一个事务。
- 应保存外部调用结果和失败原因，保证可追溯。

十四、技术关键点解决方案

请选取至少 6 个技术关键点详细展开。每个技术关键点请按：
- 背景问题
- 设计目标
- 方案设计
- 关键流程
- 优点
- 局限与后续优化
来写。

关键点1：统一 AI 工作台与 Agent 自动路由
内容：
- 用户不再手动选择智能体。
- 统一总智能体入口。
- AgentRouter 根据用户输入判断任务类型。
- 路由到 ContentAgent、KnowledgeAgent、StudentAgent、SchoolAgent、FeishuAgent。
- 工具能力通过 AgentExtension 注入上下文。
- 好处是降低用户操作成本，也便于后续扩展。

关键点2：LLM API 可替换封装
内容：
- 使用 LLMClient 抽象。
- MockLLMClient 用于无 Key 或离线 Demo。
- OpenAiCompatibleLLMClient 用于 DeepSeek 或兼容 OpenAI 协议的模型。
- 后续可接豆包、通义千问、OpenAI、本地模型网关。
- 支持流式输出。
- 支持解析 reasoning_content / reasoning / thinking 字段。
- 不把业务逻辑写死在模型客户端中。

关键点3：SSE 流式回答、模型 Thinking 与工具 Trace 展示
内容：
- SSE 适合单向流式文本输出。
- 后端返回 session、thinking、trace、reasoning、delta、done 等事件。
- delta 是最终回答。
- reasoning 是模型返回的 Thinking。
- trace 是工具调用结果。
- 前端实时展示，完成后可折叠查看。
- 后端保存到会话 metadata。
- 兼容旧会话 metadata 为空。

关键点4：工具扩展/MCP/Skill 解耦设计
内容：
- AgentExtension 统一描述扩展能力。
- McpToolProvider 表示外部工具能力。
- SkillProvider 表示固定工作流或 SOP 能力。
- RuntimeToolExtensions 支持时间、联网等能力。
- LarkCliMcpExtension 支持飞书工具。
- SemanticVectorSearchExtension 支持向量检索。
- 小组成员可在 contrib/capability 中新增 ToolService、Controller、Extension。
- 主流程稳定，降低多人协作冲突。

关键点5：知识库向量检索方案
内容：
- MySQL 保存原始知识文档。
- 文档切分为 Chunk。
- EmbeddingService 生成向量。
- Weaviate 保存向量分片。
- 查询时先向量检索。
- 若 Weaviate 不可用或无结果，fallback 到 MySQL 搜索。
- 检索结果带 source、title、chunkText、feishuToken、feishuUrl、tags、distance。
- Agent 调用向量检索后把结果注入 Prompt。
- 避免知识库无依据时胡编乱造。

关键点6：飞书生态接入方案
内容：
- 飞书文档、飞书多维表格、飞书任务、飞书机器人能力统一封装。
- Demo 阶段部分使用 Mock，部分通过 lark-cli 调用真实飞书。
- FeishuAgent 负责飞书相关任务。
- 飞书共享文件夹通过 folderToken 访问。
- 创建文档时需要 parentToken 或目标文件夹 token。
- 飞书能力调用结果进入工具 Trace。
- 后续可替换为正式飞书 OpenAPI Client。
- 权限不足时要明确提示用户授权范围。

关键点7：内容运营数据沉淀设计
内容：
- 内容生成不只是一次性输出。
- 主题库保存选题。
- 文案库保存多个版本文案。
- 内容日历保存计划和发布状态。
- 文案图片或配图建议单独管理。
- 支持 5 星评分。
- 支持使用状态和使用日期。
- 支持内容生成流水，保存输入输出摘要。
- 后续可同步飞书多维表格和文档。

关键点8：安全配置与敏感信息管理
内容：
- API Key 不放公共配置。
- 使用 application-local.yml 或环境变量。
- application-local.yml 不进入 Git。
- 前端不持有 LLM Key 或飞书 Key。
- 后端统一代理外部 API。
- 飞书和 Shell 工具使用白名单。
- 后续使用 JWT、RBAC、审计日志。
